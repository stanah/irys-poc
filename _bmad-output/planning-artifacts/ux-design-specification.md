# UX Design Specification irys-poc

**Author:** stanah
**Date:** 2026-02-12

---

## Executive Summary

### Project Vision

irys-pocは「クリエイターとファンの関係を永続的に守る」設計思想を核に、分散型プロトコルを水平統合する"薄い統合レイヤー"型動画プラットフォームのPoCである。「分散型YouTube」ではなく「Web3ネイティブのクリエイター直接支援プラットフォーム」として、Day 1からの収益化・100%透明な送金・コンテンツの永続的所有権を提供する。

UX設計の最上位原則は「Web3であることを意識させないシンプルな体験」。技術的複雑さを抽象化し、一般ユーザーがYouTubeと同程度の操作感で利用開始できることを目指す。

### Target Users

**プライマリ:**

| ペルソナ | ユウキ（22歳・ファン） | ケン（28歳・クリエイター） |
|---------|----------------------|-------------------------|
| Web3リテラシー | 低（MetaMask未所持） | 高（MetaMask所持） |
| 認証パス | AA（Googleログイン） | MetaMask直接接続 |
| コア動機 | 推しを直接支えたい | 初日から報われる場所 |
| 成功の瞬間 | 投げ銭が100%届いたことをオンチェーンで確認 | 最初の投げ銭を受け取り、手数料ゼロを確認 |
| デバイス | モバイル中心 | デスクトップ中心 |

**セカンダリ:** アクティブファン（字幕・推薦等の貢献者）— Phase 2

**将来スコープ:** コラボレーター（Revenue Split）— Phase 2、ディレクター/マネージャー（B2B）— Phase 3

### Key Design Challenges

1. **Web3オンボーディングの壁の解消** — MetaMask未所持ユーザーがウォレットの存在を意識せず、3ステップ以内でオンボーディングを完了し動画視聴を開始できること。AA（Account Abstraction）によるメール/SNSログインが鍵
2. **非同期処理の待機体験** — 投げ銭のオンチェーン確認（10-30秒）、限定動画の復号（目標3秒以内）、アップロードパイプライン（5段階）。ブロックチェーン特有の待機時間をユーザーにとって不安でなく「信頼性の証」として伝える体験設計
3. **エラーリカバリーの設計** — 残高不足、トランスコード失敗、AA失敗、ネットワークエラー等。各エラーに対して「何が起きたか・残高は安全か・次に何をすべきか」を明確に伝え、代替手段（MetaMaskフォールバック等）への誘導を含める
4. **2つの認証パスの統合** — AA（Web2ライク）とMetaMask（Web3ネイティブ）の共存。UI分岐を最小化しつつ、両方のユーザー体験を一貫させる統一Walletインターフェース
5. **限定動画アクセス制御のUX** — ロック画面→条件表示→条件達成→復号ローディング→再生という複雑なフローを、ユーザーが「特別感」として楽しめる体験に

### Design Opportunities

1. **「100%届いた」の感動演出** — オンチェーントランザクション確認のリアルタイムUI可視化。Polygonscanリンクによる透明性の体感。既存プラットフォーム（YouTube Super Chat 30%手数料）との対比が自然に伝わるデザイン
2. **関係性の可視化** — 支援履歴のオンチェーン永続記録を活かし、「初期サポーター」「累計支援額」等のファン-クリエイター関係性を見える化。ファンのアイデンティティ形成に寄与
3. **プログレッシブ・ディスクロージャー** — Web3の複雑さを段階的に開示。初期は「ログイン→視聴→投げ銭」のシンプル体験、慣れたらセルフカストディ移行・詳細なオンチェーン情報へのアクセスを提供
4. **待機時間の価値転換** — ブロックチェーン処理中のローディングUI（⏳ トランザクション処理中...、🔓 コンテンツを復号化しています...）を、単なるスピナーではなく「何が起きているか」を教育的に伝える機会として活用

## Core User Experience

### Defining Experience

irys-pocのコア体験は3層のヒエラルキーで構成される:

#### Foundation Experience（土台）— 動画視聴

ユーザーが最も頻繁に行うアクション。ここが「YouTubeと遜色ない」レベルでなければ、Signature Experienceへの到達機会すら生まれない。

- **公開動画:** URLアクセス → 1秒以内に再生開始（ログイン不要）
- **限定動画:** ACC検証 → 復号 → 3秒以内に再生開始
- **グレースフルデグラデーション:** Lit Protocol障害時も公開動画は正常再生。プラットフォーム全体が死なない安心感を設計で担保

#### Signature Experience（差別化）— 投げ銭100%到達確認

「なぜこのプラットフォームを使うのか」への回答。プラットフォームの存在意義そのもの。

- **ファン（ユウキ）:** 「0.01 ETHを送信 → ✅ TechMaster Ken に 0.01 ETH が届きました → Polygonscanで確認」— 「これだ。これが欲しかったんだ。」
- **クリエイター（ケン）:** 「🎉 ユウキさんから 0.01 ETH の投げ銭を受け取りました → ウォレットに全額入金確認」— 「たった1人のファンでも、直接つながれるんだ。」

**初回投げ銭 vs リピート投げ銭の体験分離:**

| 体験 | 初回投げ銭 | 2回目以降 |
|------|----------|----------|
| 心理的状態 | 躊躇、不安（「お金を払う」行為） | 慣れ、気軽さ |
| UI設計 | 少額プリセットを最上位に表示（0.001 ETH ≈ 数百円）。「なぜ100%届くのか」を簡潔に説明する教育的モーメントを挟む | プリセット1タップ → 送信で完了 |
| 成功後の演出 | 到達確認を丁寧に演出。Polygonscanリンクで透明性を体感させる | 到達確認は簡潔に。「累計支援額」の成長を見せる |

#### Growth Experience（成長）— 関係性の深化

リピート利用の動機。ファンとクリエイターの関係性が蓄積される体験。

- **限定動画アンロック:** 支援条件を満たすと特別なコンテンツにアクセスできる「特別感」
- **初期サポーター証明:** 支援履歴がオンチェーンに永続記録。「古参ファン」としてのアイデンティティ形成
- **支援者一覧:** クリエイター側から見た支援者との関係性の可視化（P1）

### Platform Strategy

| 項目 | 方針 | 根拠 |
|------|------|------|
| **プライマリ** | デスクトップWeb（レスポンシブ） | PoC段階はデスクトップファーストで設計・検証 |
| **モバイル** | レスポンシブ対応（最低限） | スマホユーザーが多数派である現実は認識しつつ、PoCでは機能動作の検証を優先 |
| **ブラウザ** | Chrome/Firefox/Safari/Edge 最新2バージョン | PRD Browser Matrix準拠 |
| **ネイティブアプリ** | スコープ外（Phase 3） | PRD明記 |
| **オフライン** | 非対応 | 分散型プロトコルへのリアルタイム接続が前提 |
| **特殊統合** | MetaMask拡張 + Privy AA SDK + HLS再生 | 2つの認証パスとストリーミング再生が必須 |

**モバイル対応方針（PoC段階）:**
- レイアウトはレスポンシブグリッド（既存実装の`grid-cols-1 sm:grid-cols-2 md:grid-cols-3 lg:grid-cols-4`を維持）
- タッチ操作は自然に動作する範囲で対応（タップターゲットサイズ44px以上）
- モバイル固有の最適化（ジェスチャー操作、モバイル動画プレーヤーUI等）はPost-MVP

### Effortless Interactions

**ゼロ思考で完了すべきアクション:**

| アクション | 目標 | 実現手段 |
|-----------|------|---------|
| ログイン（ファン） | Googleボタン1クリック → 即利用開始 | Privy AA（Passkey + ソーシャルログイン） |
| 公開動画の視聴 | URLアクセス → 即再生（ログイン不要） | Irys Gateway → HLS直接再生 |
| 投げ銭の送信（リピート） | プリセット1タップ → 送信で完了 | TippingWidget プリセットUI |
| ログイン（クリエイター） | MetaMask接続ボタン → 署名 → 完了 | Viem直接接続 |

**自動化すべき処理（ユーザー介入不要）:**

| 処理 | 自動化内容 |
|------|----------|
| トランスコード | アップロード後にLivepeerが自動実行。ユーザーは進捗を見るだけ |
| 暗号化 | 限定動画アップロード時にLitが自動暗号化。ユーザーはアクセスタイプを選ぶだけ |
| アクセス条件判定 | 限定動画ページで条件充足を自動検知し、自動アンロック |
| ガス代処理 | AAユーザーはPaymasterが代払い。投げ銭額＝届く額に見える |
| 収益分配 | スマートコントラクトが投げ銭受領時に自動分配（Phase 2 Revenue Split） |

### Critical Success Moments

**Make-or-Break瞬間（優先度順）:**

| 優先度 | 瞬間 | ユーザー | 成功体験 | 失敗時の影響 |
|--------|------|---------|---------|-------------|
| 1 | **初回ログイン** | ファン | 「あれ、ウォレットとか作らなくていいの？」 | 永久離脱 |
| 2 | **初回動画再生** | ファン | 「本当にWeb3なの？」と感じるほどスムーズ | 「YouTubeで十分」と判断 |
| 3 | **初回投げ銭の送信決断** | ファン | 少額プリセットと「100%届く」説明で心理的ハードルを超える | 投げ銭が発生せず、Signature Experienceに到達しない |
| 4 | **投げ銭100%到達確認** | ファン | 「全額届いた！」をオンチェーンで確認 | プラットフォームの存在意義消失 |
| 5 | **初回アップロード完了** | クリエイター | 5段階の進捗が完了し、動画が公開される | 「使いにくい」で離脱 |
| 6 | **初回投げ銭受領** | クリエイター | 手数料ゼロを確認。直接つながる実感 | 使い続ける理由消失 |
| 7 | **限定動画アンロック** | ファン | 条件達成 → 復号 → 再生の「特別感」 | 支援動機の消失 |

### Experience Principles

irys-pocの全UXデザイン判断を導く5原則:

1. **段階的透明性** — デフォルトはシンプル表示（投げ銭額のみ、処理中メッセージのみ）。詳細を見たいユーザーには「詳細を見る」でガス代・トランザクション情報・ブロック確認状況を展開。Web3の複雑さを隠すのではなく、必要な人に必要な深さで見せる
2. **Web3は手段、体験はWeb2以上** — 技術的複雑さは抽象化するが、Web3ならではの価値（永続性、100%到達、オンチェーン証明）は積極的に体験として提供する
3. **待機は不安ではなく期待** — ブロックチェーン処理の待機時間を「何が起きているか」「なぜ安全か」を伝える教育的体験に変換する
4. **エラーは行き止まりではなく迂回路** — 常に代替手段を提示。AAが失敗すればMetaMask、ネットワークエラーなら再試行。ユーザーを決してブロックしない。Lit障害時も公開動画は正常動作するグレースフルデグラデーション
5. **1人のファンでも特別** — 大規模プラットフォームの数の論理ではなく、クリエイターとファンの1対1の関係性を大切にする設計
